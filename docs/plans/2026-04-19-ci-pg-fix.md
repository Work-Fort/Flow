# Plan: Wire Postgres into the Release workflow's test job

## Overview

Master tip `81e4e2b` produced a green `CI` workflow run (`24634073285`) but a red `Release` workflow run (`24634073284`). The failure is on the `Release` workflow's `test` job, which runs `mise run ci`. Per `.mise/tasks/ci`, that target depends on `lint`, `test`, and `e2e`. The `test` task probes Postgres via `psql` against `FLOW_DB` and exits non-zero when unreachable (silent-skip elimination, per `feedback_no_test_failures.md`). The `Release` job has no Postgres service container and no `FLOW_DB` env, so it fails immediately with:

```
ERROR: FLOW_DB (postgres://postgres@127.0.0.1/flow_test?sslmode=disable) not reachable.
```

The `CI` workflow's `lint-and-unit` job already does this correctly (Postgres service + `FLOW_DB` env), and its `e2e` matrix job sets `FLOW_E2E_PG_DSN`. The fix is to bring `Release.yaml`'s `test` job up to the same wiring so `mise run ci` (which calls both `test` and `e2e`) has both DSNs available.

## Bug context

- **Failing run:** `Release` workflow, run id `24634073284`, job `test`, step `Run mise run ci`.
- **Root cause:** the `Release.test` job invokes `mise run ci` but provides no Postgres service container and no DSN env vars. The Foundation push hardened `mise run test` to require `FLOW_DB` (commit `a59e5f2`), so any caller of `mise run ci` without PG now fails fast.
- **Working precedent:** `.github/workflows/ci.yaml` `lint-and-unit` (PG + `FLOW_DB`) and `e2e` matrix (PG + `FLOW_E2E_PG_DSN`).

## Cross-repo consistency check (resolved before drafting)

The team-lead asked whether Hive's `ci` job needs PG, or whether Flow's PG-mandatory `mise run test` is the outlier.

- **Hive** (`hive/lead/.github/workflows/ci.yaml`): `ci` job runs `mise run ci` with no PG service. Hive's `mise run test` is a plain `go test ./...` with no PG probe; PG is only required by Hive's `mise run e2e`, which the `e2e-postgres` job runs separately.
- **Combine** (`combine/lead/.github/workflows/ci.yaml`): same pattern as Hive.
- **Flow** is intentionally stricter: `mise run test` mandates PG. This is consistent with the Foundation-step decision recorded in `.mise/tasks/test` ("silent skips on missing PG are forbidden").

**Decision: keep Flow's stance (option (a) in the bug context).** Flow's `mise run test` requires PG by design; the fix is to give every CI caller of `mise run ci` (or `mise run test`) a reachable Postgres. We do **not** loosen the `mise run test` contract to match Hive/Combine, because:

1. The Foundation work explicitly hardened it to fail loudly.
2. Flow's domain logic has both SQLite and Postgres stores; coverage requires the PG path to actually run.
3. The `e2e` task already supports both backends (`--backend sqlite|postgres`); demoting `test` would split coverage signals.

## Prerequisites

- Authenticated `gh` for verifying CI runs after push.
- No code changes outside `.github/workflows/release.yaml`.

## Task breakdown

### Task 1: Add Postgres service + DSN env to the Release workflow's test job

**Files:**
- Modify: `.github/workflows/release.yaml:11-19`

**Step 1: Inspect the current `test` job**

Read lines 11-19 of `.github/workflows/release.yaml`. Confirm the job is exactly:

```yaml
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0
      - uses: jdx/mise-action@v3
      - run: mise run ci
```

Expected: matches verbatim. If not, stop and report drift.

**Step 2: Replace the `test` job block**

Replace the block from `  test:` through `      - run: mise run ci` with:

```yaml
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:17
        env:
          POSTGRES_DB: flow_test
          POSTGRES_PASSWORD: flow
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 5s
          --health-timeout 5s
          --health-retries 5
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0
      - uses: jdx/mise-action@v3
      - run: mise run ci
        env:
          FLOW_DB: postgres://postgres:flow@127.0.0.1/flow_test?sslmode=disable
          FLOW_E2E_PG_DSN: postgres://postgres:flow@127.0.0.1/flow_test?sslmode=disable
```

Notes:
- The Postgres service block is copy-pasted verbatim from `.github/workflows/ci.yaml` `lint-and-unit` (same image, same `POSTGRES_DB`, same password, same health-check, same port mapping).
- Both `FLOW_DB` and `FLOW_E2E_PG_DSN` are set on the `mise run ci` step because that target depends on both `test` (uses `FLOW_DB`) and `e2e` (uses `FLOW_E2E_PG_DSN` when `--backend postgres`; the default `mise run e2e` invocation in `ci` is sqlite, but setting both keeps the env consistent and future-proofs against the `e2e` task picking up postgres mode).
- `fetch-depth: 0` is preserved because downstream jobs (`tag`, `build`, `release`) depend on full git history for tag computation.

**Step 3: Validate the YAML locally**

Run: `python3 -c "import yaml,sys; yaml.safe_load(open('.github/workflows/release.yaml'))" && echo OK`

Expected: prints `OK`. If a `YAMLError` surfaces, fix indentation (services blocks must align under the job, two spaces deeper than `test:`).
