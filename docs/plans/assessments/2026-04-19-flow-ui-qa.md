### 5. Sample data render — FAIL

Multiple frontend-backend type mismatches found. The UI will silently show empty data for three screens.

**Agent pool screen** (`/v1/agents`):

Backend `agentResponse` JSON fields: `id`, `name`, `team_id`, `model`, `runtime`, `current_role`, `current_project`, `current_workflow_id`, `lease_expires_at`.

Frontend `AgentEntry` type (`web/src/stores/agents.ts:5`): `id`, `name`, `model`, `runtime`, `assigned`, `project_id`, `role`, `workflow_id`, `lease_expires_at`.

Mismatches:
- `current_project` → frontend reads `project_id` (field missing — always undefined)
- `current_role` → frontend reads `role` (field missing — always undefined)
- `current_workflow_id` → frontend reads `workflow_id` (field missing)
- Backend has no `assigned` boolean field → frontend `agent.assigned` always undefined (controls "Claimed"/"Idle" badge)

Additionally, `createAgentsStore` calls `flowAPI.get<AgentEntry[]>('/v1/agents...')` but `/v1/agents` returns `{"agents": [...]}` not a bare array. The store returns an object where `agents()` is a single object, not an array — `For each={agents()!}` iterates nothing meaningful.

**Audit log screen** (`/v1/audit`):

Backend `auditEventResponse` JSON fields: `id`, `occurred_at`, `type`, `agent_id`, `agent_name`, `workflow_id`, `role`, `project`.

Frontend `AuditEvent` type (`web/src/stores/audit.ts:5`): `id`, `event_type`, `project_id`, `workflow_id`, `agent_id`, `metadata`, `created_at`.

Mismatches:
- `type` → frontend reads `event_type` (undefined — badge renders empty)
- `occurred_at` → frontend reads `created_at` (undefined — date shows "Invalid Date")
- `project` → frontend reads `project_id` (undefined — project filter display blank)

Additionally, `createAuditStore` calls `flowAPI.get<AuditEvent[]>('/v1/audit...')` but `/v1/audit` returns `{"events": [...]}` not a bare array. The events store returns an object; `For each={events()!}` iterates nothing.

**Work item screen** (`/v1/projects/{id}/work-items`):

Backend `workItemResponse` JSON fields: `id`, `instance_id`, `title`, `description`, `current_step_id`, `assigned_agent_id`, `priority` (string), `fields`, `created_at`, `updated_at`.

Frontend `WorkItem` type (`web/src/stores/work-items.ts:6`): `id`, `title`, `status`, `assigned_agent`, `priority` (number), `instance_id`, `updated_at`, `created_at`.

Mismatches:
- `current_step_id` → frontend reads `status` (undefined — step badge never renders)
- `assigned_agent_id` → frontend reads `assigned_agent` (undefined — agent display blank)
- `priority` is `string` in backend, typed `number` in frontend (wrong type, display undefined for numeric comparisons)

The `/v1/projects/{id}/work-items` endpoint itself returns a bare array (confirmed live: `[]`), matching the store's `WorkItem[]` expectation — so the array wrapper is correct for this endpoint. The field naming mismatches remain.

**Projects and Bots:** Shape matches. `GET /v1/projects` returns bare array. `GET /v1/projects/{id}` and bot endpoints return flat objects that match frontend types.

## Defects found

### Defect 1 — CRITICAL: Agent pool screen broken (field names + response shape)

**Area:** Area 5, `web/src/stores/agents.ts` + `internal/daemon/rest_agents.go`

**What's wrong:**
1. Backend returns `{"agents": [...]}` (wrapped). Frontend calls `flowAPI.get<AgentEntry[]>(...)` expecting a bare array. The store receives an object; the `<For>` loop gets no items.
2. Backend fields `current_project`, `current_role`, `current_workflow_id` differ from frontend fields `project_id`, `role`, `workflow_id`.
3. Backend has no `assigned` boolean; frontend uses it as the primary "claimed vs idle" gate.

**Fix:** In `web/src/stores/agents.ts`, unwrap the response: change `flowAPI.get<AgentEntry[]>(...)` to extract `.agents`. Rename frontend `AgentEntry` fields to match backend: `project_id` → `current_project`, `role` → `current_role`, `workflow_id` → `current_workflow_id`. Derive `assigned` from `lease_expires_at != null`. Update `agent-pool-view.tsx` accordingly.

---

### Defect 2 — CRITICAL: Audit log screen broken (field names + response shape)

**Area:** Area 5, `web/src/stores/audit.ts` + `internal/daemon/rest_audit.go`

**What's wrong:**
1. Backend returns `{"events": [...]}` (wrapped). Frontend calls `flowAPI.get<AuditEvent[]>(...)` expecting a bare array. Same shape problem as Defect 1.
2. Backend field `type` vs frontend `event_type`; `occurred_at` vs `created_at`; `project` vs `project_id`.

**Fix:** In `web/src/stores/audit.ts`, unwrap: extract `.events`. Align `AuditEvent` interface: rename `event_type` → `type`, `created_at` → `occurred_at`, `project_id` → `project`. Update `audit-log-view.tsx` to read `event.type` and `event.occurred_at` and `event.project`.

---

### Defect 3 — MAJOR: Work item screen — three field names wrong

**Area:** Area 5, `web/src/stores/work-items.ts` + `rest_types.go workItemResponse`

**What's wrong:** `status` → backend is `current_step_id`. `assigned_agent` → backend is `assigned_agent_id`. `priority` typed as `number` in frontend but is a `string` in backend.

**Fix:** Rename frontend `WorkItem.status` → `current_step_id`, `assigned_agent` → `assigned_agent_id`, `priority: number` → `priority: string`. Update `work-item-list.tsx` and `work-item-detail.tsx` accordingly.

---

### Defect 4 — MINOR: SPA deep-link fallback missing from `/ui/*` static server

**Area:** Area 1, `internal/daemon/ui_routes.go`

**What's wrong:** `registerUIRoutes` uses `http.FileServer` directly. A request for `/ui/projects` (a route that exists in the SPA but not as a file) returns 404 instead of serving `index.html`. For a Module Federation remote this is unlikely to be hit in practice (the Scope shell loads the module, not the URL), but direct deep-links or browser refresh on a sub-route will fail.

**Fix:** Wrap the file server to fall back to `dist/index.html` on 404, as Sharkfin does in its frontend handler. Intercept the 404 response and serve index.html instead.

### R2-4: `TestUI_UnknownPathFallsBackToIndexHTML` — genuine test, not a tautology

The test (`ui_routes_test.go:87`) uses a `fstest.MapFS` containing only `remoteEntry.js` and `index.html` — no `some/spa/route` file. It requests `/ui/some/spa/route` and asserts HTTP 200 with body containing "SPA". If the fallback were absent (i.e. the request passed to `http.FileServer`), the file server would return 404 because the path does not exist in the map FS. The test would fail. It is not a tautology.

Note: `mustSub` returns the FS as-is without calling `fs.Sub`. This is consistent with the production path which passes `fs.Sub(web.Dist, "dist")` already pre-rooted at the dist directory; the test map FS is equivalently pre-rooted.
