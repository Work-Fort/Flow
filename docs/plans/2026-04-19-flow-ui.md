# Flow Web UI — SolidJS, Embedded in Scope

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to
> implement this plan task-by-task.

## Overview

Flow now has projects, project-bots, vocabularies, work items, the
scheduler, the audit event store, the Sharkfin chat adapter, the
Combine webhook receiver, and 15 MCP tools. What it does NOT have is
any UI. The agent-pool initiative is blocked at "create the 5 dogfood
projects" because there is no UI in which an operator can drive that
work — `AGENT-POOL-REMAINING-WORK.md` calls Flow UI out as the new
critical-path prerequisite for bot bootstrap.

This plan adds the Flow web UI. Six screens:

1. **Project CRUD** — list, create, edit, delete projects.
2. **Bot identity management** — per-project bot row, Passport key ID
   display, Hive role assignments, "rotate key" affordance (stub for
   v1: opens an instructions panel; rotation is operator-driven via
   Passport UI).
3. **Work-item viewer** — list + detail for a project's work items.
4. **Agent-pool view** — live state of pool agents (claimed vs idle,
   who claimed, current work item). Read-only.
5. **Audit log viewer** — Flow's event log, filterable per project /
   per work item / per bot.
6. **Audit retention controls** — per-project setting "retention:
   permanent | N days". Stored in the project row; **no purge daemon
   in v1** (the field is recorded intent only).

Stack: SolidJS + `@workfort/ui-solid` (Lit/Lion under the hood) +
`@workfort/auth`, all consumed as Module Federation singletons from
the Scope shell. The UI is a federated remote, served by Flow's Go
backend behind the `ui` build tag, and rendered as a tab in the
Scope Tauri shell. Sharkfin's `web/` directory is the structural
template — this plan mirrors it and explicitly does NOT mirror the
Nexus UI.

## Hard constraints (non-negotiable)

These come from explicit user decisions in the autonomous-execution
session and the user-instruction memory; renegotiating them is a
plan failure.

- **Stack: SolidJS + `@workfort/ui-solid` + `@workfort/auth`.** No
  React, no Vue, no Svelte, no plain Lit, no framework adapter shim.
  The `@workfort/ui` (Lit/Lion) layer is consumed indirectly through
  `@workfort/ui-solid`'s Solid-flavoured wrappers. Web components
  from `@workfort/ui` (e.g. `<wf-banner>`, `<wf-skeleton>`) are
  acceptable inline JSX exactly as Sharkfin uses them.
- **NO per-service styling.** No Tailwind. No UnoCSS in the remote
  (the Scope shell has UnoCSS, but the remote does not). No
  application-level CSS overrides. No custom layout primitives. The
  Scope shell provides the entire visual language; Flow consumes
  `@workfort/ui-solid` components as-is. If a needed component is
  missing, raise it as a dependency on the UI library (Task 14
  documents the protocol) — do NOT hand-roll it locally.
- **Sharkfin UI is the gold-standard reference.** Mirror its
  structure exactly: Module Federation via `@module-federation/vite`
  exposing `./index` from `./src/index.tsx`, `solid-js` +
  `@workfort/ui` + `@workfort/ui-solid` as singletons,
  `vite-plugin-solid`, dual `web/embed.go` + `web/embed_ui.go` files
  keyed by `//go:build ui` build tag, `//go:embed all:dist`. The
  existing Sharkfin files at
  `~/Work/WorkFort/sharkfin/lead/web/{vite.config.ts,package.json,
  embed.go,embed_ui.go,index.html,src/index.tsx}` are the model.
- **Nexus UI is the cautionary tale** — do NOT pull patterns from
  there.
- **Embed in Scope as a tab** via Module Federation. The remote
  exposes `mount` / `unmount` / `manifest` per the
  `ServiceModule` contract documented at
  `~/Work/WorkFort/scope/lead/docs/frontend/service-contract.md`.
  The shell loads it via `loadRemote('flow/index')` — same path
  Sharkfin already takes.
- **Served by Flow's Go backend** via the `skills/lead/go-frontend-
  embed/` pattern, with one mandatory deviation: the build tag is
  **`ui`** (matching Sharkfin), NOT `spa` (which the skill
  documents). The skill is wrong on this point in our constellation;
  the Sharkfin convention wins. Task 14 amends the skill so future
  services don't have to re-learn this.
- **Pre-1.0 commit hygiene applies to the implementation phase**:
  multi-line conventional commits via HEREDOC, body explaining the
  why, `Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>`
  trailer. **No `!` markers, no `BREAKING CHANGE:` footers.**
- **No new Go runtime dependencies for the embed.** Flow already has
  `embed` in stdlib and the Sharkfin `frontend` pattern is small
  enough to lift inline (Task 4). Don't pull in the
  `scope/lead/go/frontend` package — it's not published, and copying
  the ~70 lines of file-server + cache-control logic keeps the Flow
  binary self-contained.
- **Hexagonal where it matters.** New REST handlers introduced for

- **Dual-backend e2e** per `feedback_e2e_dual_backend.md`. Every new

  ships SQLite + Postgres SQL in lock-step.
- **E2E harness independence** per
  `feedback_e2e_harness_independence.md`. The harness extension

- **No `t.Parallel()`** in e2e tests (daemon spawn cost ~200 ms).

## Open design questions resolved

These were explicitly delegated to the planner. Each answer is
chosen to minimise bootstrap clicks (the user's stated optimisation
target) but with the failure-mode tradeoff stated.

### Q1: Sharkfin channel auto-provisioning

**Resolution: AUTO.** When the operator submits the create-project
form, Flow's `POST /v1/projects` handler calls
`ChatProvider.CreateChannel(ctx, channel_name, public=true)` after
the project row commits and BEFORE the response returns. The
operator does not visit Sharkfin's UI.

**Tradeoff.** If Sharkfin is unreachable at create time, the
project row is already committed and the operator sees a 502 from
Flow. Recovery: the operator re-submits the form with
`channel_already_exists=true` (a new `bool` field on
`CreateProjectInput`), which the handler interprets as "skip
CreateChannel". For an unreachable Sharkfin the operator can
finish in two attempts: first attempt creates the project, second
attempt skips the channel. We accept this over either (a) doing
nothing (every operator must context-switch to Sharkfin) or (b)
rolling back the project row on Sharkfin failure (bot vocabulary
already permits projects without a working channel — the
`ChatProvider == nil` audit-only path is exercised in tests).

The handler logs a warning when channel creation fails and emits an
audit event of a new type `project_channel_create_failed` so the UI
can surface a banner on the next list refresh.

### Q2: Passport API key auto-minting

**Resolution: AUTO with operator-confirmed reveal.** When the
operator submits the create-bot form (which is wired into the
project-create flow as a follow-on step the UI navigates to
automatically), Flow's `POST /v1/projects/{id}/bot` handler calls
Passport's `POST /api/v1/api-keys` to mint a `wf-svc_*` key, stores
the SHA-256 hash + the returned key ID on the bot row, writes the
plaintext into `FLOW_BOT_KEYS_DIR/<bot_id>` (mode 0600), and
returns the plaintext **once** in the create-bot response body.
The UI displays the plaintext in a confirmation modal with a "copy
to clipboard" button and a "I have saved it" acknowledgement
checkbox; the modal cannot be dismissed until acknowledged. After
dismissal, no UI surface ever shows the plaintext again — `GET
/v1/projects/{id}/bot` returns only the key ID.

**Tradeoff (secret exposure).** The plaintext is in (a) Flow's
response body to the operator, (b) the FLOW_BOT_KEYS_DIR file,
(c) the operator's clipboard for as long as the OS holds it.
That's the minimum exposure for an auto-mint flow that doesn't
require a second tool. The alternative — operator manually mints
in Passport UI then pastes — moves the plaintext through a longer
chain (Passport → operator-eyes → operator-paste → Flow → file)
and adds three context switches per project, with no real
improvement in confidentiality.

**Tradeoff (Passport unreachable).** Same recovery as Q1: a
`bring_your_own_key=true` field on `CreateBotInput`. When set, the
handler skips the Passport mint call and requires the operator to
supply `passport_api_key` + `passport_api_key_id` directly (the
existing v1 contract from the bot vocabulary plan). The UI form
exposes this as an "advanced" disclosure with a warning that the
operator must mint the key themselves.

**Rotation.** The bot identity management screen exposes a
"Rotate key" action that calls a new `POST
/v1/projects/{id}/bot/rotate-key` endpoint. The endpoint mints a
new key in Passport, swaps the hash + key ID on the bot row, writes
the new plaintext to the FLOW_BOT_KEYS_DIR file (overwriting), and
returns the new plaintext one time. The Passport revocation of the
old key is best-effort (logged on failure but does not block the
swap; a Passport admin can sweep stale keys). UI shows the same
one-time-reveal modal.

### Q3: Audit retention controls UX

**Resolution: stored intent only, NO purge job.** The project row
gains a `retention_days *int` column (`NULL` = permanent). The UI
exposes a select with options `Permanent | 30 days | 90 days |
180 days | 365 days | Custom...`. On save, the value is persisted.
There is **no scheduled job** that purges audit events in v1.

The audit log viewer screen displays the configured retention
prominently with a `<wf-banner variant="info">` reading "Retention
is recorded but not yet enforced — events are stored
permanently. A future release will add scheduled purging." This is
an honest disclosure that the UI tracks intent. Failing to surface
this honestly leads downstream operators to assume purging works.

**Why this answer.** The user explicitly stated the
retention-policy decision was "permanent for v1, controls deferred
to UI". Adding the UI control without the daemon does the smaller
half of the work and surfaces the deferred half so it doesn't get
forgotten. Adding the purge daemon would expand scope into a whole
sub-workstream (transactional purge, foreign-key implications,
backup-retention-vs-purge interaction) that the user has not
authorised.

### How Scope consumes the remote

The Scope shell's `web/shell/src/lib/remotes.ts` enumerates Pylon
service-discovery results, filters where `s.ui == true`, and
registers `name: 'flow', entry: '/forts/<fort>/api/flow/ui/
remoteEntry.js'`. The shell then `loadRemote('flow/index')` which
calls our `mount(el, { connected })`. We satisfy the
`ServiceModule` contract:

```ts
export interface ServiceModule {
  mount(el: HTMLElement, props: { connected: boolean }): void;
  unmount(el: HTMLElement): void;
  manifest: { name: 'flow'; label: 'Flow'; route: '/flow';
              display: 'nav' };
}
```

Note: Scope's documented `ServiceModule` (in
`docs/frontend/service-contract.md`) describes a `default` export
returning a render function plus optional `SidebarContent` /
`HeaderActions` slots. Sharkfin's actual implementation uses
`mount` / `unmount` / `manifest` instead — Sharkfin's pattern is
what works in production today, so we follow Sharkfin verbatim. The
mismatch between docs and Sharkfin is a Scope-team concern, not
ours; do NOT attempt to bring Sharkfin (or this remote) into line
with the doc — that's a separate plan.

### How Flow's Go backend serves it

A new `internal/daemon/ui_routes.go` (Task 4) registers
`/ui/health` and `/ui/*` on the existing mux. The handler is the
inlined Sharkfin pattern (~70 lines) consuming `web.Dist` and
returning a JSON manifest. Wired in `server.go` after the existing
`registerProjectRoutes` call so it sits among the other route
groups.

The `/ui/*` namespace does NOT pass through Passport auth
middleware — see `internal/daemon/middleware.go:publicPathSkip`
for the existing pattern. The remote's `client.ts` is responsible
for attaching the `Authorization: Bearer <jwt>` header on REST
calls (the JWT comes from the `@workfort/auth` shared singleton,
which the Scope shell is already authenticated against).

### Schema change

`workflow_instances.project_id TEXT REFERENCES projects(id)` —
nullable. The bot vocabulary plan deferred this coupling explicitly;
the work-item viewer needs it. Existing rows get NULL and remain
discoverable via the legacy `team_id` filter. Migration is
non-destructive: SQLite uses `ALTER TABLE ADD COLUMN`, Postgres uses
the same syntax.

### Domain change

`domain.Project.RetentionDays *int` — nullable pointer. NULL =
permanent. Stored as `INTEGER NULL` on both backends.

### Hive client extension

The new `HiveAgentClient` interface today has only Claim / Release /
Renew. The agent-pool view needs ListAgentsByAssignment. We extend
the interface and the Hive adapter (Task 5). The Hive Go client
already exposes `ListAgentsByAssignment` (see
`hive/lead/client/agents.go:108`), so the adapter is a thin wrap.

```go
// SPDX-License-Identifier: GPL-2.0-only
package daemon

import (
    "encoding/json"
    "io/fs"
    "net/http"
    "strings"
)

// uiManifest is the JSON envelope the Scope shell decodes from
// /ui/health. Mirrors the Sharkfin shape so the shell handles both
// services with one ServiceTracker code path.
type uiManifest struct {
    Status  string   `json:"status"`
    Name    string   `json:"name"`
    Label   string   `json:"label"`
    Route   string   `json:"route"`
    WSPaths []string `json:"ws_paths"`
}

const (
    uiName  = "flow"
    uiLabel = "Flow"
    uiRoute = "/flow"
)

// registerUIRoutes mounts /ui/health and the static file server at
// /ui/* against fsys. fsys must be rooted at the Vite output dir
// (the caller passes fs.Sub(web.Dist, "dist") for the embedded
// case, or a testing fs in tests). When fsys has no remoteEntry.js,
// /ui/health returns 503 — Scope's ServiceTracker treats that as
// "service reachable, UI not built".
func registerUIRoutes(mux *http.ServeMux, fsys fs.FS) {
    hasRemoteEntry := fileExists(fsys, "remoteEntry.js")

    mux.HandleFunc("GET /ui/health", func(w http.ResponseWriter, r *http.Request) {
        w.Header().Set("Content-Type", "application/json")
        m := uiManifest{Name: uiName, Label: uiLabel, Route: uiRoute, WSPaths: []string{}}
        if hasRemoteEntry {
            m.Status = "ok"
            w.WriteHeader(http.StatusOK)
        } else {
            m.Status = "unavailable"
            w.WriteHeader(http.StatusServiceUnavailable)
        }
        _ = json.NewEncoder(w).Encode(m)
    })

    fileServer := http.StripPrefix("/ui/", http.FileServer(http.FS(fsys)))
    mux.Handle("/ui/", http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        path := strings.TrimPrefix(r.URL.Path, "/ui/")
        if strings.HasPrefix(path, "assets/") {
            w.Header().Set("Cache-Control", "public, max-age=31536000, immutable")
        } else {
            w.Header().Set("Cache-Control", "no-cache")
        }
        fileServer.ServeHTTP(w, r)
    }))
}

func fileExists(fsys fs.FS, name string) bool {
    f, err := fsys.Open(name)
    if err != nil {
        return false
    }
    _ = f.Close()
    return true
}
```

```go
// HiveAgentFilter mirrors the optional query params on Hive's
// /v1/agents endpoint. Empty values omit the corresponding filter.
type HiveAgentFilter struct {
    TeamID     string
    Assigned   string // "" | "true" | "false"
    WorkflowID string
    Role       string
    Project    string
}

// HiveAgentRecord is a flattened view of a Hive Agent + its current
// pool assignment, sufficient for the agent-pool view. Adapter maps
// this from the Hive client's Agent type. Field names mirror Hive's
// `client.Agent` verbatim (see hive/lead/client/types.go:24-36) so
// the adapter is a straight identity copy and there is no risk of
// the developer transposing Hive's mixed Assigned/Current vocabulary
// into a uniform-but-wrong Assigned-everything shape.
type HiveAgentRecord struct {
    ID                string
    Name              string
    TeamID            string
    Model             string
    Runtime           string
    AssignedRole      string     // Hive: AssignedRole
    CurrentProject    string     // Hive: CurrentProject (note: Current, not Assigned)
    CurrentWorkflowID string     // Hive: CurrentWorkflowID (note: Current, not Assigned)
    LeaseExpiresAt    *time.Time // nil when idle; pointer because encoding/json does not honour omitempty for zero time.Time
}

// HiveAgentClient extension:
type HiveAgentClient interface {
    ClaimAgent(ctx context.Context, role, project, workflowID string, ttlSeconds int) (*HiveAgent, error)
    ReleaseAgent(ctx context.Context, id, workflowID string) error
    RenewAgentLease(ctx context.Context, id, workflowID string, ttlSeconds int) error
    ListAgents(ctx context.Context, filter HiveAgentFilter) ([]HiveAgentRecord, error)
}
```

```go
// ListAgents implements domain.HiveAgentClient by proxying to
// hiveclient.ListAgentsByAssignment and copying each Agent's flat
// fields into a HiveAgentRecord. Hive's `client.Agent` already
// has flat fields (no nested Assignment struct) — see
// hive/lead/client/types.go:24-36. Idle agents arrive with a zero
// LeaseExpiresAt; we convert that to a nil *time.Time so JSON
// callers see no field instead of "0001-01-01T00:00:00Z" (which
// the UI's lease-countdown would render as a wildly negative
// duration).
func (a *Adapter) ListAgents(ctx context.Context, filter domain.HiveAgentFilter) ([]domain.HiveAgentRecord, error) {
    agents, err := a.client.ListAgentsByAssignment(ctx,
        filter.TeamID, filter.Assigned, filter.WorkflowID,
        filter.Role, filter.Project)
    if err != nil {
        return nil, fmt.Errorf("hive list agents: %w", err)
    }
    out := make([]domain.HiveAgentRecord, 0, len(agents))
    for _, ag := range agents {
        rec := domain.HiveAgentRecord{
            ID:                ag.ID,
            Name:              ag.Name,
            TeamID:            ag.TeamID,
            Model:             ag.Model,
            Runtime:           ag.Runtime,
            AssignedRole:      ag.AssignedRole,
            CurrentProject:    ag.CurrentProject,
            CurrentWorkflowID: ag.CurrentWorkflowID,
        }
        if !ag.LeaseExpiresAt.IsZero() {
            t := ag.LeaseExpiresAt
            rec.LeaseExpiresAt = &t
        }
        out = append(out, rec)
    }
    return out, nil
}
```

```go
    Offset     int       `query:"offset" minimum:"0"`

type AuditFilteredOutput struct {
    Body struct {
        Events []auditEventResponse `json:"events"`

func registerAuditRoutes(api huma.API, store domain.AuditEventStore) {

        Method:      http.MethodGet,
        Path:        "/v1/audit",

            EventType:  in.EventType,
            Since:      in.Since,
            Until:      in.Until,
            Limit:      in.Limit,
            Offset:     in.Offset,
        })
```

    ```sql
    -- Adds nullable project_id FK to workflow_instances. SQLite's

    -- (limitation of SQLite ALTER), so the column is plain TEXT and
    ```

    ```sql
    -- 005_workflow_instance_project_fk.sql (PostgreSQL)
    -- Adds nullable project_id FK to workflow_instances. PG supports
    ```

```ts
// SPDX-License-Identifier: GPL-2.0-only
import { useAuth } from '@workfort/ui-solid';

const BASE = ''; // relative — same-origin via Scope's reverse proxy

async function request<T>(method: string, path: string, body?: unknown): Promise<T> {
  const { token } = useAuth();
  const headers: Record<string, string> = { 'Content-Type': 'application/json' };
  const tk = token();
  if (tk) headers.Authorization = `Bearer ${tk}`;
  const res = await fetch(BASE + path, {
    method,
    headers,
    body: body === undefined ? undefined : JSON.stringify(body),
  });
  if (!res.ok) {
    const text = await res.text().catch(() => '');
    throw new FlowAPIError(res.status, text || res.statusText);
  }
  if (res.status === 204) return undefined as T;
  return (await res.json()) as T;
}

export class FlowAPIError extends Error {
  constructor(public status: number, message: string) { super(message); }
}

export const flowAPI = {
  get: <T>(path: string) => request<T>('GET', path),
  post: <T>(path: string, body?: unknown) => request<T>('POST', path, body),
  patch: <T>(path: string, body?: unknown) => request<T>('PATCH', path, body),
  delete: <T>(path: string) => request<T>('DELETE', path),
};
```
